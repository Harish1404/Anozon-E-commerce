1. Permission Before Modification
Always ask for explicit approval before:
Editing existing files
Refactoring large sections
Deleting or renaming files
Clearly explain:
What will change
Why it’s needed
Expected impact
🧠 2. Understand Before Acting
Analyze the full context before making changes
Do not assume missing details — ask clarifying questions
Respect existing architecture and conventions
🧹 3. Write Clean & Maintainable Code
Follow:
Single Responsibility Principle
Clear naming conventions
Modular structure
Avoid unnecessary complexity
Prefer readability over cleverness

🧪 5. Ensure Code Quality
Validate:
Syntax correctness
Type safety (if applicable)
Edge cases
Never introduce breaking changes silently

📚 6. Explain Changes Clearly
Provide:
What was changed
Why it was changed
Any trade-offs
Keep explanations concise and developer-focused

🔐 7. Security First
Never expose:
Tokens
Secrets
Sensitive data
Follow best practices for:
Authentication
Validation
Data handling

⚡ 8. Performance Awareness

Avoid unnecessary re-renders, loops, or API calls
Suggest optimizations only when meaningful
Do not over-optimize prematurely

🔄 9. Consistency with Tech Stack
Follow the project’s:
Framework patterns (e.g., Next.js, FastAPI)
State management approach
Folder structure
Do not introduce new libraries without justification

🧱 10. Safe Refactoring
Refactor only when:
It improves clarity or maintainability
It does not break existing functionality

🚫 11. Avoid Assumptions
Do not:
Invent APIs
Guess database schema
Assume business logic

Ask instead

🔁 13. Iterative Approach
Prefer:
Small, incremental changes
Step-by-step improvements
Avoid large, risky rewrites
